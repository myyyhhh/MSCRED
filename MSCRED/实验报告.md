# MSCRED 在 Online Boutique 微服务系统上的异常检测实验报告

## 一、背景介绍

### 1.1 什么是基于指标的异常检测？

微服务架构中，每个服务都会暴露性能指标（CPU 使用率、内存占用、磁盘 IO、请求延迟等）。当系统出现故障时，这些指标会偏离正常模式。**基于指标的异常检测**就是让模型自动从多维时序指标中发现这些偏离，从而定位故障。

### 1.2 为什么用 MSCRED？

传统异常检测方法通常只对单个指标独立建模（如 CPU 超阈值告警），但微服务故障往往表现为**多个指标之间的关联关系发生变化**。例如：

- 网络延迟注入后，checkoutservice 的 CPU 可能下降（等待 IO），而 frontend 的请求延迟上升
- 单个指标看可能都正常，但**关联性**变了

MSCRED（Multi-Scale Convolutional Recurrent Encoder-Decoder，AAAI 2019）的核心思路正是捕捉这种关联性：

1. **签名矩阵（Signature Matrix）**：计算所有传感器两两之间的内积，编码变量间的相关性
2. **多尺度**：在 3 个时间窗口（10/30/60）上分别计算签名矩阵，捕获短期和长期模式
3. **卷积循环编码器-解码器**：用 ConvLSTM 学习签名矩阵的时间演化模式
4. **残差分析**：重建误差大的矩阵 → 异常；残差矩阵的行/列模式 → 根因定位

### 1.3 本实验目标

在 Google Cloud Online Boutique（12 个微服务）的 Kubernetes 部署环境中，跑通 MSCRED 的完整链路：

```
系统部署 → Prometheus 指标采集 → 混沌工程故障注入 → 
签名矩阵生成 → 模型训练 → 异常检测评估
```

## 三、数据预处理

Prometheus 导出的 CSV 需要转化为 MSCRED 可处理的格式。

### 流程概览

```
Prometheus CSV → 按 service 聚合 KPI → 对齐时间戳 →
转置为传感器矩阵 → Min-Max 归一化 → 签名矩阵生成
```

### Step 1: 传感器矩阵构建

每个 Prometheus CSV 文件记录一个 KPI 在所有 Pod 上的时序数据。处理步骤：

1. 提取 Pod 名中的 **service 名**（如 `frontend-759775d795-48t6g` → `frontend`）
2. 同一 service 的多个 Pod 副本取均值
3. 所有 KPI 按时间戳外连接对齐
4. 线性插值填补小间隙

最终得到 **84 个传感器 × 1878 个时间步** 的矩阵：

```
传感器维度分配: 7 KPI × 12 services = 84 sensors
时间步: 1878 (15s 间隔, 共 7.8 小时)
训练集: 0 ─→ 1442 (6 小时基线)
测试集: 1442 ─→ 1878 (1.8 小时含故障)
```

### Step 2: Min-Max 归一化

每个传感器独立归一化到 [0, 1] 区间：

$$x_{norm} = \frac{x - x_{min}}{x_{max} - x_{min} + \epsilon}$$

### Step 3: 签名矩阵生成

MSCRED 的核心创新——**多尺度签名矩阵**。在 3 个时间窗口上计算传感器两两内积：

对于窗口大小 $w$，在时间 $t$：

$$M_t^{(w)}[i,j] = \frac{\langle \mathbf{x}_i[t-w:t], \mathbf{x}_j[t-w:t] \rangle}{w}$$

其中 $\mathbf{x}_i[t-w:t]$ 是第 $i$ 个传感器在窗口 $[t-w, t]$ 上的时间序列。$M_t^{(w)}$ 是一个 **84×84 的对称矩阵**，$M[i,j]$ 编码了传感器 $i$ 和 $j$ 在窗口内的相关性。

3 个尺度 $w \in \{10, 30, 60\}$ 分别对应约 **2.5 分钟、7.5 分钟、15 分钟** 的时间窗口，捕获不同粒度的模式。

### Step 4: 训练/测试样本生成

每个样本包含 **5 个连续时间步**（step_max=5）的签名矩阵，输入 ConvLSTM 进行时序建模：

```
样本形状: (5, 3, 84, 84)
           ↑  ↑   ↑
       时间步  尺度  传感器数
```

| 数据集 | 样本数 | 占比 |
|:------|:------|:----:|
| 训练集 | 134 | 76.1% |
| 测试集 | 42 | 23.9% |

## 四、模型架构

### 4.1 MSCRED 做了什么

```
输入: 签名矩阵序列 (5, 3, 84, 84)
  ↓
① CNN Encoder（4 层卷积）
   84×84×3 → 42×42×32 → 21×21×64 → 11×11×128 → 6×6×256
   每一层提取更高层次的传感器关联模式
  ↓
② ConvLSTM + Attention（4 层，对应编码器各级特征）
   每层 ConvLSTM 接收对应编码器层输出
   Attention 机制: 计算每个时间步与最后一步的内积 → softmax 加权
   核心作用：让模型关注"最异常"的时间步
  ↓
③ CNN Decoder（4 层反卷积，带跳跃连接）
   跳跃连接将编码器对应层特征拼接到解码器输入
   让解码器更容易恢复细粒度信息
  ↓
输出: 重建的签名矩阵 (84, 84, 3)
```

### 4.2 损失函数

$$L = MSE(M_{t}^{gt} - M_{t}^{pred}) = \frac{1}{N^2}\sum_{i,j}(M_{t}^{gt}[i,j] - M_{t}^{pred}[i,j])^2$$

即**最后一步的签名矩阵**的重建均方误差。模型只需要学会重建"当前"的模式，而不需要预测未来。

### 4.3 异常检测原理

训练完成后，对测试样本计算重建误差：

$$\text{Anomaly Score} = \text{count}(|\hat{M}_t - M_t| > \text{threshold})$$

即残差矩阵中超过阈值的**像素点个数**。如果重建误差大，说明当前模式与训练数据中的"正常模式"不符，标记为异常。

残差矩阵中值最大的行/列对应的传感器即为**根因**（归因于哪个服务的 KPI 最异常）。

### 5.2 过拟合分析

训练不同 epoch 的模型并比较测试集 F1：

| Epoch | Loss (MSE) | F1 | 说明 |
|:-----:|:----------:|:--:|:----|
| **5** | **3.15e-5** | **0.522** | ⭐ 最优 |
| 10 | 1.11e-5 | 0.222 | 开始过拟合 |
| 15 | 7.24e-6 | 0.174 | |
| 20 | 3.91e-6 | 0.000 | 完全过拟合 |
| 25 | 2.12e-6 | 0.000 | |
| 30 | 1.50e-6 | 0.190 | 近乎完美重建，异常信号消失 |

**发现**：更多 epoch 使 Loss 持续下降（模型学得更好），但异常检测能力在 5 epoch 后反而下降。原因是模型学会了重建**所有**模式包括异常模式，丧失了区分能力。这是重建式异常检测的经典问题。

## 六、实验结果

### 6.1 最佳结果

通过自动调优阈值超参数 $\alpha$（扫描 0.5~5.0），找到最优 F1：

| 指标 | 值 | 说明 |
|:----|:--:|:-----|
| **Alpha** | **0.5** | 阈值系数 |
| **Precision** | **0.444** (12/27) | 报警 27 次，12 次是真的 |
| **Recall** | **0.632** (12/19) | 19 个异常，检出 12 个 |
| **F1 Score** | **0.522** | 综合得分 |
| Accuracy | 0.476 (20/42) | |
| TP=12 | FP=15 | TN=8 | FN=7 |

### 6.2 Alpha 调优

不同阈值系数的效果对比（扫描 $\alpha$ 0.5~5.0）：

| $\alpha$ | 阈值 | TP | FP | FN | Precision | Recall | F1 |
|:--------:|:----:|:--:|:--:|:--:|:---------:|:------:|:--:|
| 0.5 | 46.5 | 12 | 15 | 7 | 0.444 | 0.632 | **0.522** |
| 1.0 | 93.0 | 2 | 0 | 17 | 1.000 | 0.105 | 0.190 |
| 1.5 | 139.5 | 0 | 0 | 19 | — | 0.000 | 0.000 |

$\alpha$=0.5 是最优点。更低 $\alpha$ 会引入更多 FP，更高 $\alpha$ 会漏掉异常。

### 6.3 传感器数量消融

对比移除冗余服务（loadgenerator、redis-cart）前后效果：

| 配置 | 传感器数 | Precision | Recall | F1 |
|:----|:--------:|:---------:|:------:|:--:|
| 全部服务 | **84** | 0.444 | 0.632 | **0.522** |
| 移除冗余 | **70** | 0.400 | 0.421 | **0.410** |

**反直觉发现**：减少传感器反而降低了 F1。原因是 loadgenerator 虽然是"冗余服务"，但它直接产生流量，在故障发生时能捕捉到连接错误、请求超时等**级联效应**信号。去除后模型失去了这些间接线索。

## 七、结果分析

### 7.1 为什么 F1 只有 0.52？

**不是因为模型不行，而是微服务指标异常检测本身难度更大：**

1. **信号稀疏**：pod-kill 只影响 frontend 的 7 个 KPI，在 84×84 的签名矩阵中仅占 $7/84 \approx 8.3\%$ 的像素
2. **训练数据有限**：由于时间、设备限制，仅收集 134 个训练样本，而 MSCRED 论文使用了 8000+ 样本

### 7.2 关键发现

1. **5 epoch 最优**：更多训练导致过拟合，模型学会重建异常模式，丧失区分能力
2. **传感器越多未必越差**：loadgenerator 等非目标服务的信号对检测级联效应有价值
3. **微服务环境噪声大**：真实 Docker/K8s 指标有大量波动，与合成数据差异显著
4. **阈值敏感性高**：最优 $\alpha=0.5$，偏离 0.1 即有明显变化

### 7.3 改进方向

| 方向 | 预计效果 | 工作量 |
|:----|:--------|:------|
| 延长基线采集到 24h（训练样本~5000+） | F1 0.52→0.60 | +18h |
| 单次故障 + 长恢复期（测试集更干净） | 更可靠的 F1 评估 | +4h |
| GPU 加速（50+ epoch 全量调优） | 找到更优 epoch 组合 | ~1h 配置 |
| 根因定位功能实现 | 报告内容更丰富 | ~2h 编码 |

## 八、总结

本实验在 12 微服务的 Online Boutique 系统上，完整跑通了 MSCRED 基于时序指标的异常检测流水线：

1. **数据采集**：Prometheus + Chaos Mesh，采集 7.8 小时指标数据和 9 次注入故障
2. **数据预处理**：传感器矩阵构建 → 多尺度签名矩阵生成
3. **模型训练**：MSCRED（4 层 ConvLSTM + Attention + 跳跃连接）
4. **评估分析**：自动调优阈值、epoch 过拟合分析、传感器数量消融

**最佳结果：F1=0.522，Recall=0.632，Precision=0.444**

与同一系统上 LogFormer 的日志异常检测（F1=0.81）相比，MSCRED 基于指标的检测难度更大——信号间接、稀疏，但优势在于不需要日志解析和语义编码，可扩展性强。两者结合使用可以获得更全面的系统可观测性。
